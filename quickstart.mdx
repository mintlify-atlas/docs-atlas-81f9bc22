This guide walks you through cloning the project, configuring your environment, and executing tests against the Place APIs using Cucumber and REST Assured.

## Prerequisites

Before you begin, ensure the following tools are installed and available on your `PATH`:

<CardGroup cols={2}>
  <Card title="Java 11+" icon="coffee">
    The framework requires Java 11 or later. Verify with `java -version`. OpenJDK and Oracle JDK are both supported.
  </Card>
  <Card title="Maven 3.x" icon="box">
    All build, test, and report tasks are driven by Maven. Verify with `mvn -version`.
  </Card>
  <Card title="Git" icon="git-branch">
    Required to clone the repository. Verify with `git --version`.
  </Card>
  <Card title="IDE (recommended)" icon="code">
    IntelliJ IDEA with the Cucumber for Java plugin provides Gherkin syntax highlighting, step navigation, and in-IDE test execution.
  </Card>
</CardGroup>

## Steps

<Steps>
  <Step title="Clone the repository">
    Clone the project to your local machine and navigate into the project root:

    ```bash
    git clone https://github.com/your-org/bdd-api-framework.git
    cd bdd-api-framework
    ```

    The project layout after cloning:

    ```
    src/
    ├── main/java/pojo/
    │   ├── AddPlace.java
    │   └── Location.java
    └── test/java/
        ├── cucumber/Options/TestRunner.java
        ├── features/placeValidations.feature
        ├── resources/
        │   ├── APIResources.java
        │   ├── TestDataBuild.java
        │   ├── Utils.java
        │   └── global.properties
        └── stepDefinitions/
            ├── Hooks.java
            └── StepDefinition.java
    ```
  </Step>

  <Step title="Configure global.properties">
    All tests read their base URL from `src/test/java/resources/global.properties`. Open the file and confirm the `baseURL` value:

    ```properties
    baseURL=https://rahulshettyacademy.com
    ```

    This property is loaded at runtime by `Utils.java` using `FileInputStream`. The default value points to the live training API and requires no change for standard test runs.

    <Warning>
      `Utils.java` contains a hardcoded Windows absolute path when constructing the `FileInputStream` for `global.properties` (e.g., `C:\\Users\\...\\global.properties`). This will cause a `FileNotFoundException` on macOS and Linux.

      Replace the hardcoded path with a path relative to the project root so the file resolves correctly on any OS:

      ```java
      // Before (Windows-only, breaks on macOS/Linux)
      FileInputStream fis = new FileInputStream(
          "C:\\Users\\YourName\\project\\src\\test\\java\\resources\\global.properties"
      );

      // After (cross-platform, resolves from the Maven working directory)
      FileInputStream fis = new FileInputStream(
          "src/test/java/resources/global.properties"
      );
      ```

      Maven always executes from the project root directory, so the relative path `src/test/java/resources/global.properties` resolves correctly on all operating systems.
    </Warning>
  </Step>

  <Step title="Install dependencies">
    Download all Maven dependencies declared in `pom.xml` without running any tests:

    ```bash
    mvn dependency:resolve
    ```

    Key dependencies pulled in during this step:

    | Dependency | Version | Purpose |
    |---|---|---|
    | `rest-assured` | 6.0.0 | HTTP request construction and response validation |
    | `cucumber-java` | 7.33.0 | Gherkin step definition binding |
    | `cucumber-junit` | 7.33.0 | JUnit runner integration for Cucumber |
    | `jackson-databind` | 3.0.3 | POJO serialization for request bodies |
    | `groovy` | 3.0.25 | GPath expression support in REST Assured assertions |
  </Step>

  <Step title="Run all tests">
    Execute the full test suite. Maven compiles sources, then runs the JUnit `TestRunner`, which discovers all feature files under `src/test/java/features`:

    ```bash
    mvn test
    ```

    `TestRunner.java` is configured with the following `@CucumberOptions`:

    ```java
    @RunWith(Cucumber.class)
    @CucumberOptions(
        features = "src/test/java/features",
        plugin   = "json:target/jsonReports/cucumber-report.json",
        glue     = {"stepDefinitions"}
    )
    public class TestRunner {}
    ```

    - **`features`** — points to `placeValidations.feature`, which contains the `@AddPlace` and `@DeletePlace` scenarios.
    - **`glue`** — binds to `StepDefinition.java` and `Hooks.java` in the `stepDefinitions` package.
    - **`plugin`** — writes a Cucumber JSON report to `target/jsonReports/cucumber-report.json` after each run.

    A successful run prints a summary table to the console:

    ```
    Tests run: 2, Failures: 0, Errors: 0, Skipped: 0
    ```
  </Step>

  <Step title="Run tests by tag">
    Cucumber tags let you target a specific scenario or group of scenarios without modifying the feature file.

    <Tip>
      Pass the `-Dcucumber.filter.tags` system property to `mvn test` to run only the scenarios matching a given tag expression. Tags defined in `placeValidations.feature` are `@AddPlace`, `@DeletePlace`, and `@Regression`.
    </Tip>

    Run only the add-place scenario:

    ```bash
    mvn test -Dcucumber.filter.tags="@AddPlace"
    ```

    Run only the delete-place scenario:

    ```bash
    mvn test -Dcucumber.filter.tags="@DeletePlace"
    ```

    Run all scenarios tagged `@Regression` (currently both scenarios):

    ```bash
    mvn test -Dcucumber.filter.tags="@Regression"
    ```

    Combine tags with boolean operators:

    ```bash
    # Run scenarios tagged @AddPlace OR @DeletePlace
    mvn test -Dcucumber.filter.tags="@AddPlace or @DeletePlace"

    # Run scenarios tagged @Regression but NOT @DeletePlace
    mvn test -Dcucumber.filter.tags="@Regression and not @DeletePlace"
    ```

    <Note>
      The `@DeletePlace` tag triggers a `@Before` hook in `Hooks.java` that pre-populates a `place_id` in the scenario context before the scenario steps execute. Running the delete scenario in isolation requires the target API to already contain a valid place created in a prior run, or the hook will handle creation automatically.
    </Note>
  </Step>

  <Step title="Generate the HTML report with mvn verify">
    The `maven-cucumber-reporting` plugin (version 5.9.0) is bound to the `verify` phase. Running `mvn verify` executes tests **and** generates an interactive HTML report from the Cucumber JSON output:

    ```bash
    mvn verify
    ```

    After the build completes, the report is written to:

    ```
    target/
    ├── jsonReports/
    │   └── cucumber-report.json   ← raw Cucumber JSON
    └── cucumber-html-reports/
        └── overview-features.html ← open this in a browser
    ```

    Open `target/cucumber-html-reports/overview-features.html` in any browser to view scenario results, step-level pass/fail status, and execution duration charts.
  </Step>

  <Step title="Interpret the output">
    After a test run, three output artifacts are available:

    **Console output**

    Cucumber prints each scenario and step result directly to the terminal. Passed steps are shown in green, failed steps in red with the assertion error message and stack trace inline.

    ```
    Feature: Validating Place APIs

      Scenario Outline: Verify is Place is being successfully added   # src/test/java/features/placeValidations.feature:4
        Given add Place Payload with "AHouse" "English" "World Cross Center"
        When user calls "addPlaceAPI" with "POST" http request
        Then the API call got success with status code 200
        And "status" in response body is "OK"
        And "scope" in response body is "APP"
        And verify place_Id created maps to "AHouse" using "getPlaceAPI"
    ```

    **logging.txt**

    REST Assured request and response logging is captured to `logging.txt` in the project root (written by `Utils.java` via `PrintStream`). Inspect this file to see the full HTTP traffic — request headers, body, response status, and response payload — for every API call made during the run.

    **Cucumber JSON report**

    `target/jsonReports/cucumber-report.json` contains structured machine-readable results. This file is consumed by `maven-cucumber-reporting` to produce the HTML report, and can also be ingested by CI systems (Jenkins Cucumber Reports plugin, GitLab test reports, etc.).
  </Step>
</Steps>
